hi

a few more comments about the searching-over-OAI issue ...

- i think OAI-PMH should always be directed towards harvesting - 
extensions are great, but there must be clean separation. the approach 
we advocate is to separate the archive and extension into distinct 
components. the archive then becomes the data provider, and the 
component an OAI service provider (and a data provider in the context of 
a different protocol  - this different protocol is what we're interested in)

- our solution is meant to be general and extensible (we don't want to 
invent query languages - we want to look into using familiar wrappers) 
the extensibility mechanism also must cover different searching 
strategies and, more generally, other DL requirements beyond searching...

- for a peek at the testbed we have been running for the last 8 months 
now, try http://purl.org/net/etdunion

- you can download and install a reference implementation of the search 
component from http://oai.dlib.vt.edu/odl

- more components are in the works :)
