Abstract

Open Archives Initiative Object Reuse and Exchange (OAI-ORE) defines standards for the description and exchange of aggregations of Web resources. OAI-ORE introduces the notion of a Resource Map, which is a specialization of a named graph that asserts a finite set of resources (the Aggregated Resources), their types, intra-relationships, and relationships with resources external to this finite set (the external resources). A Resource Map Document is a machine-readable representation of a Resource Map. A Resource Map Document can be serialized in different formats, and the purpose of this document is to specify a serialization based on, and compliant with the Atom syndication format. Hereby, a Resource Map Document is an Atom Feed Document with some ORE-specific ingredients. This Atom-based format to serialize Resource Map Documents may be referred to as the Resource Map Profile of Atom. This specification is one of several documents comprising the OAI-ORE specification and user guide.

1. Introduction

Open Archives Initiative Object Reuse and Exchange (OAI-ORE) defines standards for the description and exchange of aggregations of Web resources. OAI-ORE introduces the notion of a Resource Map, which is a specialization of a named graph that asserts a finite set of resources (the Aggregated Resources), their types, intra-relationships, and relationships with resources external to this finite set (the external resources). A Resource Map Document is a machine-readable representation of a Resource Map. A Resource Map Document can be serialized in different formats, and the purpose of this document is to specify a serialization based on, and compliant with the Atom syndication format [RFC4287]. Hereby, a Resource Map Document is an Atom Feed Document with some ORE-specific ingredients. This Atom-based format to serialize Resource Map Documents may be referred to as the Resource Map Profile of Atom.

This specifcation uses concepts and terms introduced in the following ORE documents:

  • ORE Specification - Abstract Data Model [ORE Model];
  • ORE Specification - Vocabulary [ORE Vocabulary].

The Resource Map Profile of Atom is fully compliant with the Atom Syndication Format [RFC4287] but defines a specific profile of it:

  • Section 2 defines how entities of the ORE Model relate to entities of the Atom model.
  • Section 3 and Section 4 restrict the use of some Atom constructs and elements, and specify how to use the extensibility features of Atom to address ORE requirements.

The ORE User Guide - Resource Map Implementation in Atom [ORE User Guide Resource Map] provides Atom implementation guidance to supplement this specification.

1.1 Notational Conventions

This specification uses the namespace prefix "atom:" for the Atom Namespace URI "http://www.w3.org/2005/Atom" as defined in Section 1.2 of [RFC4287]. Note that the choice of namespace prefix is arbitrary and not semantically significant.

This specification uses the same shorthands to terms from the XML Infoset [XML Infoset] as used in [RFC4287]: the phrase "Information Item" is omitted when naming Element Information Items and Attribute Information Items. Therefore, when this specification uses the term "element," it is referring to an Element Information Item in Infoset terms. Likewise, when it uses the term "attribute," it is referring to an Attribute Information Item.

This specification uses the common term URI [RFC3986] to mean both IRI [RFC3987] and URI.

The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC2119].

1.2 Compliance

A Resource Map Document serialized according to the Resource Map Profile of Atom MUST be compliant with the Atom Syndication Format [RFC4287]. Moreover, it MUST adhere to the semantic mapping between the ORE Model and the Atom model detailed Section 2, as well as to the restrictions and extensions of the Resource Map Profile of Atom detailed in Section 3 and Section 4.

Specifically, a Resource Map Document MUST be a well-formed XML document that is valid according to the Relax NG Compact Schema of [RFC4287]. Moreover, a Resource Map Document MUST be valid according to the more specific Schematron Schema of Appendix C that further restricts the syntax of Resource Map Profile of Atom.

1.3 Example

Throughout the remainder of this document, the following scenario will be used to illustrate the specifics of the Resource Map Profile of Atom.

D-Lib Magazine <http://www.dlib.org> decides to embrace the OAI-ORE interoperability specification by minting Resource Map URIs (URI-R) for each of its articles, and by making a Resource Map Document available from each of those URIs. A Resource Map Document lists all the Web resources of which a D-Lib article is composed. The scenario zooms in on the following article:

2. Correspondence between the ORE Model and the Atom Model

Table 1 and Figure 1 illustrate how entities from the ORE Model map to entities from the Atom model [RFC4287].

In general, an Atom feed corresponds to a Resource Map, and an Atom Feed Document corresponds to a Resource Map Document. Moreover, an Atom entry in the Atom feed exists for each Aggregated Resource of the aggregation described by the Resource Map. Each Aggregated Resource is conveyed in such an Atom entry by means of a unique atom:link child element of atom:entry that expresses a relationship with a value of "related".

More specifically, the following holds with regard to the mapping:

  • Child elements of atom:feed that are in the Atom namespace pertain to the Resource Map. The exceptions to this rule are:
    • The atom:id child element of atom:feed pertains to the Atom feed only and has no correspondence in the ORE Model;
    • The atom:link child element of atom:feed that expresses a relationship with a value of "related" pertains to the Aggregation described by the Resource Map (see below).
  • Child elements of atom:feed that are not in the Atom namespace, i.e. extension child elements of atom:feed, pertain to the Aggregation described by the Resource Map;
  • Child elements of atom:entry that are in the Atom namespace pertain to the Atom entry only; they have no correspondence in the ORE Model. The exceptions to this rule are:
    • The atom:link child element of atom:entry that expresses a relationship with a value of "alternate" is used to convey an Aggregated Resource (see below);
    • The atom:link child element of atom:entry that expresses a relationship with a value of "via" is used to convey the Lineage of an Aggregated Resource (see below).
  • Child elements of atom:entry that are not in the Atom namespace, i.e. extension child elements of atom:entry, pertain to the Aggregated Resource that is conveyed using the atom:entry;
  • Four types of atom:link elements used as a child elements of atom:feed or atom:entry make the connection between the ORE and the Atom world:
    • An atom:link child element of atom:feed that expresses a relationship with a value of "describes" conveys the URI (URI-A) of the Aggregation described by the Resource Map;
    • An atom:link child element of atom:feed that expresses a relationship with a value of "related" conveys a URI (other than URI-A) by which the Aggregation is known;
    • An atom:link child element of atom:entry that expresses a relationship with a value of "alternate" conveys the URI of an Aggregated Resource (URI-AR) of the Aggregation described by the Resource Map (URI-R);
    • An atom:link child element of atom:entry that expresses a relationship with a value of "via" conveys that the Aggregated Resource of the Aggregation described by the Resource Map is also an Aggregated Resource in another Resource Map. This atom:link element allows, for example, establishing a chain of provenance for an Aggregated Resource as it is being aggregated in different Aggregations over time.

Figure 1 illustrates this mapping. It shows a picture similar to the one illustrating the ORE Model in the ORE Abstract Model document, but has Atom entities overlaid on it. The left hand of the picture shows the Atom feed that coincides with the Resource Map, while the right hand shows the Atom entries of that Atom feed that are introduced to convey the Aggregated Resources of the Aggregation described by the Resource Map. Typed relationships from the ORE Model are shown as black arrows, whereas the constructs from the Atom serialization are shown in red arrows. When a black arrow and a red arrow carry the same number, this indicates that the specific typed relationship from the ORE Model is implemented in Atom using the construct with the same number. For example the black arrow [2] asserts <A-1> <owl:sameAs> <DOI-1> according to the ORE Model, and conveys an identifier (DOI-1) that is associated with the Aggregation described by the Resource Map. In the Atom serialization, the corresponding construct is shown as a red arrow and also carries a [2]. It indicates that the assertion from the ORE Model is expressed in Atom by means of an atom:link child element of the atom:feed that has a relationship attribute with a value of "related" and an href attribute with a value of "DOI-1". Similarly, the black arrow [3] asserts <A-1> <ore:aggregates> <AR-1> according to the ORE Model. In the Atom serialization, the corresponding construct is shown as the red arrow that also carries the [3]. It indicates that the Aggregated Resource is conveyed by means of an atom:entry that has an atom:link child element that has a relationship attribute with a value of "alternate" and an href attribute with a value of "AR-1", which is the URI of the Aggregated Resource.

3. Use of Atom Constructs in the Resource Map Profile of Atom

This Section specifies the use of common Atom Constructs in the Resource Map Profile of Atom. When, in the remainder of this document, an Atom element is identified as being a particular kind of construct, it inherits the corresponding requirements from that construct's definition in this Section 3.

3.1 Text Constructs

In the Resource Map Profile of Atom:

  • The Text construct MUST be used in a manner described in the Text Construct section of [RFC4287].

3.2 Person Constructs

In the Resource Map Profile of Atom:

  • The Person construct MUST be used in a manner described in the Person Construct section of [RFC4287].

3.3 Date Constructs

In the Resource Map Profile of Atom:

  • The Date construct MUST be used in a manner described in the Date Construct section of [RFC4287].
  • In the Resource Map Profile of Atom, expressing date-times in UTC is RECOMMENDED. As a result, the use of the special UTC designator ("Z") is RECOMMENDED over the notation using a numeric time zone offset. Hence, the RECOMMENDED format is YYYY-MM-DDThh:mm:ssZ.

4. Use of Atom Elements in the Resource Map Profile of Atom

4.1 Container Elements

4.1.1 The "atom:feed" Element

In the Resource Map Profile of Atom:

  • The "atom:feed" element has semantics and properties as described in the atom:feed Element section of [RFC4287] unless specified differently below.
  • An Atom feed corresponds with a Resource Map of the ORE Model, and an Atom Feed Document corresponds with the Resource Map Document of the ORE Model.
  • The following rules apply regarding child elements of the atom:feed element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • Child elements of atom:feed that are in the Atom namespace pertain to the Resource Map. The exceptions to this rule are:
      • The atom:id child element of atom:feed pertains to the Atom feed only and has no correspondence in the ORE Model;
      • The atom:link child element of atom:feed that expresses a relationship with a value of "related" pertains to the Aggregation described by the Resource Map (see below).
    • Child elements of atom:feed that are not in the Atom namespace, i.e. extension child elements of atom:feed, pertain to the Aggregation described by the Resource Map, i.e. Atom feed;
    • atom:feed elements MUST contain one or more atom:author elements.
      • The atom:author element conveys the authoring authority of the Resource Map.
    • atom:feed elements MUST contain exactly one atom:category element identifying the Atom Feed Document as a Resource Map Document. This element has a scheme attribute value of "http://www.openarchives.org/ore/terms/", a term attribute value of "http://www.openarchives.org/ore/terms/ResourceMap", and a label attribute value of "Resource Map".
    • atom:feed elements MAY contain any number of atom:category elements in addition to the aforementioned one.
    • atom:feed elements MAY contain any number of atom:contributor elements.
    • atom:feed elements MUST contain one or more atom:entry elements.
      • Each atom:entry element is used to convey an Aggregated Resource.
    • atom:feed elements MUST NOT contain more than one atom:generator element.
    • atom:feed elements MUST NOT contain more than one atom:icon element.
      • Use of an atom:icon element with content of "http://www.openarchives.org/ore/logos/ore_icon.png" is RECOMMENDED to allow recognition of the Atom Feed Document as a Resource Map Document.
    • atom:feed elements MUST NOT contain more than one atom:logo element.
      • Use of an atom:logo element with content of "http://www.openarchives.org/ore/logos/ore_logo.png" is RECOMMENDED to allow recognition of the Atom Feed Document as a Resource Map Document.
    • atom:feed elements MUST contain exactly one atom:id element.
      • The atom:id element has no correspondence in the ORE Model.
    • atom:feed elements MUST contain one atom:link element with a rel attribute value of "self". This conveys the preferred URI for retrieving Atom Feed Documents representing this Atom feed.
      • The href attribute value of this atom:link element MUST be the URI of the Resource Map (URI-R).
    • atom:feed elements MUST contain exactly one atom:link element with a rel attribute value of "describes". Such an atom:link element is used to convey the URI-A (= URI-R#aggregation) of the Aggregation described by the Resource Map.
      • The href attribute value of this atom:link element must be URI-A (= URI-R#aggregation) .
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "related". Such an atom:link element is used to convey URIs other than URI-A (= URI-R#aggregation) by which the Aggregation described by the Resource Map is known. The href attribute value of this atom:link element must be a URI other than URI-A by which the Aggregation described by the Resource Map is known.
    • atom:feed elements MAY contain additional atom:link elements beyond those described above.
      • These atom:link elements have no correspondence in the ORE Model.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "alternate" that has the same combination of type and hreflang attribute values.
    • atom:feed elements MUST NOT contain more than one atom:rights element.
      • It is RECOMMENDED to express the rights pertaining to the Resource Map by setting the content of the atom:rights element to the URI of a rights license.
    • atom:feed elements MUST NOT contain more than one atom:subtitle element.
    • atom:feed elements MUST contain exactly one atom:title element.
    • atom:feed elements MUST contain exactly one atom:updated element.
    • atom:feed elements MAY contain any number of child elements from namespaces other than the Atom namespace. These elements convey Types and Relationships pertaining to the Aggregation (URI-A = URI-R#aggregation) described by the Resource Map.
    • An Atom Feed Document MUST NOT have multiple atom:entry elements with the same atom:id value, as is permitted by [RFC4287]. Such entries would convey the same Aggregated Resource, and might contain conflicting assertions about that Aggregated Resource within the confines of the same Resource Map. As a result, updating an atom:entry in the Resource Map Profile of Atom MUST always be achieved by editing the existing atom:entry and by changing the value of its atom:updated child element. It MUST NOT be achieved by adding an additional atom:entry with the same atom:id as the already existing one.

4.1.2 The "atom:entry" Element

In the Resource Map Profile of Atom:

  • The "atom:entry" element has semantics and properties as described in the atom:entry Element section of [RFC4287] unless specified differently below.
  • An Atom entry conveys an Aggregated Resource and Types and Relationships pertaining to the Aggregated Resource.
  • The following rules apply regarding child elements of the atom:entry element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • Child elements of atom:entry that are in the Atom namespace pertain to the Atom entry only; they have no correspondence in the ORE Model. The exceptions to this rule are:
      • The atom:link child element of atom:entry that expresses a relationship with a value of "alternate" is used to convey an Aggregated Resource of the ORE Model (see also below);
      • The atom:link child element of atom:entry that expresses a relationship with a value of "via" is used to convey the Lineage of an Aggregated Resource, as defined in the the ORE Model (see also below).
    • Child elements of atom:entry that are not not in the Atom namespace, i.e. extension child elements of atom:entry, pertain to the Aggregated Resource that is conveyed using the atom:entry. They correspond to Types and Relationships pertaining to the Aggregated Rersource, as defined in the ORE Model.
    • atom:entry elements MUST NOT contain any atom:author elements. Authorship information is inherited in the following order:
      • from the atom:author element contained in an atom:source child element of the atom:entry if such atom:source element exists;
      • from the atom:author elements of the atom:feed element if no such atom:source element exists.
    • atom:entry elements MAY contain any number of atom:category elements.
      • The atom:category element to express that the atom:entry conveys an Aggregated Resource is implicit and can be derived from the fact that the atom:feed of which the atom:entry is a child element has an atom:category that types it as a Resource Map.
    • atom:entry elements MUST NOT contain any atom:content elements.
    • atom:entry elements MAY contain any number of atom:contributor elements.
    • atom:entry elements MUST contain exactly one atom:id element.
      • The atom:id element has no correspondence in the ORE Model.
      • The content of this atom:id element MUST NOT be the URI of the Aggregated Resource (URI-AR).
    • atom:entry elements MUST contain one atom:link element with a rel attribute value of "alternate". This link conveys the Aggregated Resource.
      • The href attribute value of this atom:link element MUST be the URI of the Aggregated Resource (URI-AR).
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "via". This link conveys the Lineage of the Aggregated Resource.
      • The href attribute value of this atom:link element MUST be the URI of a /Resource Map with which the Aggregated Resource has a Lineage relationship.
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "self". This is the preferred URI for retrieving Atom Entry Documents representing this Atom entry.
      • The href attribute value of this atom:link element MUST NOT be the URI of the aggregated resource (URI-AR).
    • atom:entry elements MUST NOT contain any atom:link elements other than the aformentioned ones.
    • atom:entry elements MUST NOT contain more than one atom:published element.
    • atom:entry elements MUST NOT contain any atom:rights elements. Rights information is inherited in the following order:
      • from the atom:rights element contained in an atom:source child element of the atom:entry if such atom:source element exists;
      • from the atom:rights elements of the atom:feed element if no such atom:source element exists.
    • atom:entry elements MUST NOT contain more than one atom:source element.
    • atom:entry elements MUST NOT contain more than one atom:summary element.
    • atom:entry elements MUST contain exactly one atom:title element.
    • atom:entry elements MUST contain exactly one atom:updated element.
    • atom:entry elements MAY contain any number of child elements from namespaces other than the Atom namespace. These elements convey Types and Relationships pertaining to the Aggregated Resource (URI-AR) conveyed by the Atom entry.

4.2.7 The "atom:link" Element

In the Resource Map Profile of Atom:

  • The "atom:link" element has semantics and properties as described in the atom:link Element section of [RFC4287] unless specified differently below.
  • The "atom:link" element makes the connection between the Atom and the ORE world. It is used to convey:
    • The URI of the Aggregation (URI-A=URI-R#aggregation) described by the Resource Map.
    • URIs (other than URI-A=URI-R#aggregation) that are associated with the Aggregation described by the Resource Map;
    • The URIs of the Aggregated Resources of the Aggregation described by the Resource Map;
    • The URIs of other Resource Maps that also aggregate an Aggregated Resource of the Aggregation described by the Resource Map (URI-R) at hand.
  • The atom:link element has attributes as described in the sections The "href" Attribute, The "rel" Attribute, The "type" Attribute, The "hreflang" Attribute, The "title" Attribute and The "length" Attribute of [RFC4287].
  • The following rules apply regarding the atom:link element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST contain one atom:link element with a rel attribute value of "self". This conveys the preferred URI for retrieving Atom Feed Documents representing this Atom feed.
      • The href attribute value of this atom:link element MUST be the URI of the Resource Map (URI-R);
      • The rel attribute value MUST be "self";
      • The type attribute value MUST be "application/atom+xml".
    • atom:feed elements MUST contain exactly one atom:link element with a rel attribute value of "describes". Such an atom:link element is used to convey the URI-A (= URI-R#aggregation) of the Aggregation described by the Resource Map.
      • The href attribute value of this atom:link element must be URI-A (= URI-R#aggregation) of the Aggregation described by the Resource Map.
      • The rel attribute value MUST be "describes".
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "related". Such an atom:link element is used to convey URIs other than URI-A (= URI-R#aggregation) by which the Aggregation described by the Resource Map is known.
      • The href attribute value of this atom:link element must be a URI other than URI-A by which the Aggregation described by the Resource Map is known;
      • The rel attribute value MUST be "related".
    • atom:feed elements MAY contain additional atom:link elements beyond those described above.
      • These atom:link elements have no correspondence in the ORE Model.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "alternate" that has the same combination of type and hreflang attribute values.
    • atom:entry elements MUST contain one atom:link element with a rel attribute value of "alternate". This link conveys an Aggregated Resource.
      • The href attribute value of this atom:link element MUST be the URI of the Aggregated Resource (URI-AR).
      • The rel attribute value MUST be "alternate".
      • The type attribute SHOULD be used to express the media type of the Aggregated Resource if it is known to the authoring authority of the Resource Map.
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "via". This link conveys the Lineage of the Aggregated Resource. As explained in the atom:source section, the ORE Lineage notion is distinct from the Atom source notion.
      • The href attribute value of this atom:link element MUST be the URI of a Resource Map with which the Aggregated Resource has a Lineage relationship;
      • The rel attribute value MUST be "via";
      • The type attribute value MUST be "application/atom+xml".
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "self". This is the preferred URI for retrieving Atom Entry Documents representing this Atom entry.
      • The href attribute value of this atom:link element MUST NOT be the URI of the aggregated resource (URI-AR) that is provided in another atom:link child element of the atom:entry;
      • The rel attribute value MUST be "self";
      • The type attribute value MUST be "application/atom+xml".
    • atom:entry elements MUST NOT contain any atom:link elements other than the aformentioned ones.

In the D-Lib Magazine example, the URI of the Resource Map (URI-R) is provided as the preferred URI from which the Atom Feed Document can be retrieved. This is achieved by means of an atom:link child element of the atom:feed element with a rel attribute of "self" and a href value of "http://www.dlib.org/dlib/february06/smith/rem/".

The URI of the Aggregation described by this Resource Map (URI-A) is provided by means of an atom:link child element of the atom:feed element with a rel attribute of "describes" and a href value of "http://www.dlib.org/dlib/february06/smith/rem/#aggregation".

Other URIs (e.g. DOI) that are associated with the Aggregation described by the Resource Map Document are also provided using an atom:link element. These elements have a rel attribute value of "related" and a href attribute value set to another URI associated with the Aggregation, e.g. "info:doi/10.1045/february2006-smith".

The first entry of the Atom feed conveys the Aggregated Resource "http://www.dlib.org/dlib/february06/smith/02smith.html". The URI of this resource (URI-AR) is provided as the value of the href attribute of the atom:link child element of the atom:entry element that has a rel attribute with a value of "alternate". The approach to point at other Resource Maps that also aggregate this resource, by using an atom:link element with a rel attribute set to "via" is explained in detail in the section on the atom:source element.

This URI is provided as the value of the href attribute of the atom:link child element of the atom:entry element that has a rel attribute with a value of "self".

4.2.9 The "atom:published" Element

In the Resource Map Profile of Atom:

  • The "atom:published" element has semantics and properties as described in the atom:published Element section of [RFC4287] unless specified differently below.
  • The "atom:published" element is a Date construct indicating an instant in time associated with an event early in the life cycle of the Atom entry.
  • The "atom:published" element has no correspondence in the ORE Model.
  • The following rules apply regarding the atom:published element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:entry elements MUST NOT contain more than one atom:published element.
    • The atom:published element SHOULD indicate the date-time when an Atom entry was first created, i.e. when it was first included in the Resource Map Document. Since the reason of the creation of an Atom entry is the aggregation of a resource, this approach allows tracking back when a resource was first aggregated by the Resource Map. Note that the date-time when a resource was first created is conceptually different than the date-time when it was first included in the Aggregation described by a Resource Map.

  • The "atom:rights" element is a Text construct that conveys information about the rights pertaining to a Resource Map (Atom feed). It corresponds to the rights related to a Resource Map as defined in the ORE Model.

    • atom:feed elements MUST NOT contain more than one atom:rights element.
    • Although the Atom specification suggests that the atom:rights element should be used to convey human readable rights information, the machine-oriented use case of the Resource Map Profile of Atom justifies the following best practice approach:
      • It is RECOMMENDED to use the atom:rights element.
      • It is RECOMMENDED to provide as the content of the atom:rights element a URI identifying the rights license that pertains to the Resource Map. This could, for example, be a Creative Commons License URI.

      • from the atom:rights elements of the atom:feed element if no such atom:source element exists.

4.2.11 The "atom:source" Element

In the Resource Map Profile of Atom:

  • The "atom:source" element has semantics and properties as described in the atom:source Element section of [RFC4287] unless specified differently below.
  • The atom:source element is used to allow the re-use of Atom entries in Atom feeds other than the one in which the entry was first published, while retaining information about an Atom entry's origin Atom feed.
  • The "atom:source" element has no correspondence in the ORE Model; it is distinct from the Aggregated Resources and External Aggregations notion of the ORE Model.
  • The atom:source element provides a way to tie an Aggregated Resource to a Resource Map in which it was aggregated. This tie remains in place even when the atom:entry that conveys the Aggregated Resource is re-used in one or more atom:feed (Resource Map or other) that are not the Resource Map in which that atom:entry was first published.
  • The following rules apply regarding the atom:source element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:entry elements MUST NOT contain more than one atom:source element.
    • If an atom:entry is copied from an origin Atom Feed Document (Resource Map Document) into a target Atom Feed Document, then the origin Atom Feed Document's metadata (all child elements of atom:feed that are in the Atom namespace, other than the atom:entry elements) SHOULD be preserved within the copied entry. This is achieved by adding an atom:source child element, if it is not already present, to the atom:entry and including some or all of the origin Atom Feed Document's metadata elements as the atom:source element's children. The following child elements of the origin Atom Feed Document's (Resource Map'Document) atom:feed element MUST be preserved in this manner: atom:id, atom:link, atom:title, atom:rights, atom:category, and atom:updated.

Presume Alice creates a Resource Map in which she wants to aggregate the PDF file "http://www.dlib.org/dlib/february06/smith/pg1-13.pdf" of the D-Lib Magazine article. Alice could obviously start from scratch and create an atom:entry in her Resource Map to convey the PDF file. But, she might also have come across the Resource Map published for the article by D-Lib Magazine, and decide to re-use its atom:entry that conveys the PDF file. In this case, Alice (well, Alice's software that is) would copy the respective atom:entry from D-Lib Magazine's Resource Map Document and paste it into hers. The pasted atom:entry is identical to the one from the D-Lib Magzine Resource Map, except for the (automatic) addition of crucial metadata to the copied entry's atom:source child element (in this case the atom:author, atom:category, atom:id, atom:link, atom:update and atom:rights from the D-Lib Resource Map Document). Alice's pasted entry aggregates the PDF file, as was her intent, but can also unambiguously be traced back to its origin in the D-Lib Magazine Resource Map.

If later, Bob discovers Alice's Resource Map and wants to re-use her copy/pasted entry that conveys the PDF file in his Resource Map, he can perform a similar copy/paste operation. Note that in Bob's case the source information is already available in Alice's Resource Map and must not be overwritten. The same situation occurs when Carol re-uses the entry from Bob's Resource Map: because source information is already present, the atom:entry is copied "as is", with inclusion of the atom:source child element.

4.2.15 The "atom:updated" Element

In the Resource Map Profile of Atom:

  • The "atom:updated" element has semantics and properties as described in the atom:updated Element section of [RFC4287] unless specified differently below.
  • The "atom:updated" element is a Date construct indicating the most recent instant in time when an Atom Feed (Resource Map ) or Atom Entry was modified in a way the authoring authority considers significant. Therefore, not all modifications necessarily result in a changed atom:updated value.
  • The "atom:updated" element, when used as a child element of an Atom feed corresponds to the last modification datetimestamp of the Resource Map as defined in the ORE Model.
  • The "atom:updated" element, when used as a child element of an Atom entry, has no correspondence in the ORE Model.
  • The following rules apply regarding the atom:updated element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST contain exactly one atom:updated element.
    • atom:entry elements MUST contain exactly one atom:updated element.
    • The content of atom:updated SHOULD be set to a new value when relevant changes are made to an atom:feed or an atom:entry. Examples of such relevant changes MAY include the following:
      • the aggregation of an additional resource (i.e. the creation of a new atom:entry that acts as a container for an additional Aggregated Resource);
      • the removal of an Aggregated Resource (i.e. deletion of the atom:entry that acts as a container for the Aggregated Resource) ;
      • the addition or deletion of Resource Map Relationships and/or Resource Map Types expressed in the Resource Map;
      • the addition of URIs that are asscoiated with the Aggregation described by the Resource Map;
      • the update of an Aggregated Resource.
    • The content of atom:updated child element of atom:feed MUST be set to a new value if the atom:updated child element of one of its atom:entry elements is set to a new value.

In the D-Lib Magazine example, a Resource Map for the article was first published on August 17th of 2007, more than a year after the publication of the journal article itself. As a result, D-Lib Magazine changed the date-time of the associated atom:updated element, leaving the date-time of the associated atom:published element unchanged. The change in the atom:updated element of this one atom:entry caused D-Lib Magazine to also change the date-time of the atom:updated element of the atom:feed.

4.2.16 Use of Atom Extension Elements in the Resource Map Profile of Atom

This Section specifies how the Resource Map Profile of Atom makes use of the extension elements allowed by [RFC4287]. The use of extension elements of an atom:feed element are discussed in Section 4.2.16.1, and extension elements of an atom:entry element are discussed in Section 4.2.16.2.

4.2.16.1 Extension Elements for the "atom:feed" Element

The Resource Map Profile of Atom allows the use of OPTIONAL extension elements for the atom:feed element to convey Relationships and Types pertaining to the Aggregation described by the Resource Map. These Relationships and Types are conveyed as child elements of atom:feed that are not in the Atom namespace.

In the Resource Map Profile of Atom:

  • atom:feed elements MAY have any number of child elements from namespaces other than the Atom namespace.
  • All non-Atom child elements of atom:feed MUST assert Relationships and Types pertaining to the Aggregation described by the Resource Map, i.e. they MUST all pertain to the resource identified by URI-A (= URI-R#aggregation).
  • The content of these non-Atom child elements of atom:feed SHOULD be a URI in order to allow for unambiguous machine processing.

In the D-Lib Magazine example, the extension child elements of the atom:feed element are used to convey several Types and Relationships of the Aggregation described by the Resource Map. D. GRDDL crosswalk from Atom XML to RDF/XML

The up-to-date version of the GRDDL-compliant XSLT to transform an Atom-based serialization of a Resource Map into a serialization in RDF/XML is available at http://www.openarchives.org/ore/0.1/atom-grddl.xsl. This XSLT uses XSLT 2.0 functionality. In order to allow Atom-based Resource Maps to be readily transformed in this manner, it is RECOMMENDED to add GRDDL namespace and transformation information as attributes to the atom:feed element, as shown below.

<?xml version="1.0" encoding="UTF-8"?>
<atom:feed xmlns:atom="http://www.w3.org/2005/Atom"
      xmlns:grddl="http://www.w3.org/2003/g/data-view#"
      grddl:transformation="http://www.openarchives.org/ore/atom/atom-grddl.xsl">
...
