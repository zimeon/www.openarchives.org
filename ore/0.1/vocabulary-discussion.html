1. Introduction

This document is designed to be a staging area for vocabulary terms that have been proposed for the main OAI-ORE Vocabulary document. It lists Properties, Relationships and Classes from vocabularies other than the ORE specific vocabulary. Each entry below has the following information, and any new entries submitted should follow the same layout.

Title
A heading for the entry
Description
A free text description of when the term should be used.
Name
The name of the term in the vocabulary it comes from
URI
The URI which identifies the term, if one exists
ORE Entities
How the term should be used with the 4 primary ORE entities
Use Cases
Motivation for why this term should be included in the main vocabulary

2. Properties

The Properties in the vocabulary document are reasonably firm decisions, however exactly which entities they can be applied to is still very much up for debate. In particular:

  • As an Aggregation is tied directly to the Resource Map, should it be allowed to have different properties to the Resource Map? For example, is the creator of the Aggregation necessarily the same as the Resource Map?
  • The time of last modification for the Aggregation can certainly be different to that of the resource map, if the change was to an external entity (and hence not part of the Aggregation). Here is one property that seems like it must be able to be different.
  • Which properties should be expressed about Aggregated Resources? We don't want to get into the field of re-expressing everything in Dublin Core or more in Resource Maps, that information should be in related documents. (See rdfs:seeAlso) However some basic information would be useful for processors... what information?
  • Should we be able to express properties for External Resources at all? As above, we don't want to fill the Resource Map with potentially out of date information about resources that aren't even part of the Aggregation. However, without at least a pointer to where an agent can find more information about the resource, the identifier is potentially completely useless (if it's not de-referencable and not a known scheme, for example).

Additional Specific properties:

Format
The file format of the entity, typically a string literal containing a MIME type. Other possible format specifications include Apple's Uniform Type Identifier and Pronom's format identifier.
  • Name: dc:format
  • URI: http://purl.org/dc/elements/1.1/format

  • Aggregations MUST NOT have a format. It is an abstract object.
  • Resource Maps MAY have a format. If present, it MUST be "application/atom+xml"
  • Aggregated Resources SHOULD have a format. It SHOULD be a mimetype, or drawn from a publically accessible set of formats.
  • External Resources MAY have a format, if they are dereferencable Resources. It SHOULD be a mimetype, or drawn from a publically accessible set of formats.
Example: 
URI-AR-1 dc:format "application/pdf"
Motivating Use Cases:
  • Format is important for machine processing of files, as neither the identifier nor the transfer protocol may include this information. For example, the format of "ftp://ftp.somewhere.com/pub/dataset/allRecords" could be in any format.
Identifier
A string literal which can be used to unambiguously reference the entity within the ORE context. An object MAY have more than one identifier, but an identifier MUST identify exactly one entity. All identifiers within the OAI-ORE context MUST be URIs.
  • Name: dc:identifier
  • URI: http://purl.org/dc/elements/1.1/identifier

  • Aggregations MUST have a URI identifier. It is the URI of the Resource Map that describes it, followed by "#aggregation".
  • Resource Maps MUST have an HTTP URI identifier. It MUST NOT be the identifier for the Aggregation or any of the Aggregated Resources. The ORE specified serialisation MUST be available by dereferencing the identifier.
  • Aggregated Resources MUST have an ultimately dereferencable URI identifier. They may also have additional non URI identifiers.
  • External Resources MUST have a URI identifier.
Example: 
URI-AR-1 dc:identifier "http://www.openarchives.org/ore/vocabulary/"
Motivating Use Cases:
  • Actually, we don't need an identifier property as in order to talk about any object it MUST already have an identifier. However when serialising, the identifier for the object itself should be present (eg it should be self-identifying)... So does this deserve to be in the main vocabulary, or just a footnote?

3. Relationships

As with properties, the subject and object of many of the relationships is up for debate.

owl:sameAs
Should this be used for expressing equivalency between aggregated resources, or should dcterms:hasFormat be used, or another relationship entirely?
  • Resource Maps MAY have equivalences stated.
  • Aggregated Resources MAY have equivalences stated.
  • External Resources SHOULD NOT have equivalences stated.

And there are many additional relationships which will be important across disciplines.

Additional Metadata
The described resource has additional metadata available at, or is otherwise described by, the referenced resource.

Rather than expressing large amounts of metadata about every resource within the Resource Map, it may be desirable to point to a further resource which has this information. For example, an Aggregated Resource might be further described by a file in XML. External Resources may also have this relationship, in order to avoid making these assertions in the Resource Map.

  • Name: rdfs:seeAlso

  • Aggregations MAY have additional metadata in other resources.
  • Resource Maps MAY have additional metadata.
  • Aggregated Resources MAY have additional metadata.
  • External Resources MAY have additional metadata.

Motivating Use Cases:

  • Citeseer document has pointer to ACM equivalent.
  • External or Aggregated PDF has a lot of additional metadata known, but not suitable for presenting in the Resource Map.

Audience
The intended audience of a resource. The object should be a class of agent, either Person or SoftwareAgent. For example, a Dataset resource would be intended for a SoftwareAgent to interact with, whereas a textual description of that resource would be intended for a Person.
  • Name: dcterms:audience
  • URI: http://purl.org/dc/terms/audience

  • Aggregations MUST NOT have an intended audience. They are not experienced directly by any agent.
  • Resource Maps MUST NOT have an intended audience. It is SoftwareAgent.
  • Aggregated Resources MAY have an intended audience.
  • External Resources SHOULD NOT have an intended audience.

Motivating Use Cases:
  • The same resource may be available in different formats, one intended for human use and the other for machine interpretation. The formats may not be sufficient information to discern the difference. For example, a short description in XML with a stylesheet of a dataset may be useful to a human, whereas the full dataset in XML without a stylesheet is not.
  • A 'splash page' is the human equivalent of a Resource Map. Combined with rdfs:seeAlso (above) we can make this relationship explicit.

Conformance to Standards
A reference to an established standard to which the resource conforms.

For example, Resource Maps may conform to one or more application or domain profiles. These profiles might specify additional requirements, or simply list additional vocabularies to use.

This MUST NOT be used to denote conformance with a file format, nor give a semantic type, even if that format or type is a standard. See the Format property.

  • Name: dcterms:conformsTo
  • URI: http://purl.org/dc/terms/conformsTo

  • Aggregations MAY conform to a standard. For example a standard which designates good practices for aggregations of a certain type.
  • Resource Maps MAY conform to a standard.
  • Aggregated Resources MAY conform to a standard.
  • External Resources SHOULD NOT conform to a standard.

Motivating Use Cases:
  • Communities will produce conformance documents or profiles for recommended best practices in creatingn resource maps. ConformsTo allows the creator to assert compliance.
  • Aggregated and External resources may conform to any number of different non format/non semantic type standards. For example, a vocabulary might conform to the ISO11179 guidelines.
  • If the format is text/xml, should conformsTo be used for representing schema/dtd information, or is there a better relationship?

dcterms:hasFormat vs owl:sameAs

owl:sameAs has the semantics:

The described resource is intellectually the same as the referenced resource.

dcterms:hasFormat has the semantics:

The described resource pre-existed the referenced resource, which is essentially the same intellectual content presented in another format.

The pre-existing notion is troublesome for asserting that two Aggregated Resources are equivalent. However, it is perhaps a closer fit for this usage than owl:sameAs?


References
The described resource references, cites, or otherwise points to the referenced resource.

This is the general relationship for citation, linking and similar. If communities want to be more specific about the type of citation or reference, then the relationship should be a child of this one. Also note that the inverse relationship is frequently used as well and can easily be accommodated by switching the subject and object. For example on may wish to list works which cite a component resource, rather than works which the component resource cites.

  • Name: dcterms:references
  • URI: http://purl.org/dc/terms/references

  • Aggregations MAY have references.
  • Resource Maps MUST NOT have references.
  • Aggregated Resources MAY have references.
  • External Resources SHOULD NOT have references.

Motivating Use Cases:
  • Citations. What more motivating use case do we have in the scholarly publication world?
  • But is there a more specific and widely understood relationship that we should be using for such an important one?

Related Resources
The described resource is somehow related to the referenced resource.

This is the general relationship for any non reference relationship. For example, in order to list resources related by topic or genre.

  • Name: dc:relation
  • URI: http://purl.org/dc/elements/1.1/relation

  • Aggregations MAY have related resources
  • Resource Maps MAY have related resources.
  • Aggregated Resources MAY have related resources.
  • External Resources SHOULD NOT have related resources.

Motivating Use Cases:
  • Most splash pages have links to related or similar resources, eg Citeseer, YouTube etc.
  • By linking to known, related Resource Maps, we will improve discovery and use.

Replaces
The described resource supplants, displaces, or supersedes the referenced resource.

Similar to hasVersion, but the old resource should not be referenced again in the future. For example, a resource map that does not validate should be replaced by one that does. However a resource map where a new relationship has been added or a typo fixed may warrant a hasVersion relationship instead or as well as. The older document may also wish to be changed to specify that it is replaced by the new one.

  • Name: dcterms:replaces
  • URI: http://purl.org/dc/terms/replaces

  • Aggregations MAY replace previous versions.
  • Resource Maps MAY replace previous versions.
  • Aggregated Resources MAY replace previous versions.
  • External Resources SHOULD NOT have replaces relationships listed.

Motivating Use Cases:
  • The new resource may not be a version of the old resource. For example if an aggregation (lowercase a) is described again by a different creator to the first description, and should be used instead of it. For example, a third party could describe an important resource and then the creator might describe it more authoritatively. One would want to say that the new description replaces the old description, but is not a version of it.

Requirements
The described resource requires the referenced resource to support its function, delivery, or coherence of content.

For example, a component resource may be a dataset that requires an application to interact with it. The application may not be able to be a component, and hence the necessity to identify how the resource is to be displayed.

  • Name: dcterms:requires
  • URI: http://purl.org/dc/terms/requires

  • Aggregations MUST NOT have requirements.
  • Resource Maps MUST NOT have requirements.
  • Aggregated Resources MAY have requirements.
  • External Resources SHOULD NOT have requirements.

Motivating Use Cases:
  • Every format requires some sort of interpreting software with which to interact with it. For very common formats, we can assume that this is not necessary but for custom datasets or rare formats, it would be important for preservation and re-use to be able to reference the software required. For example, an ARFF data file requires the WEKA data mining toolkit.

Thumbnail
The referenced resource is a thumbnail version of the described resource.
  • Name: foaf:thumbnail
  • URI:

  • Aggregations are not images and hence MUST NOT be part of a thumbnail relationship.
  • Resource Maps are not images and hence MUST NOT be part of a thumbnail relationship.
  • Aggregated Resources MAY be thumbnails, if they are images.
  • External Resources MAY be thumbnails, if they are images.

Motivating Use Cases:
  • Flickr, PhotoBucket etc. Or anything that has thumbnails that need to be explicitly described rather than using owl:sameAs.

Version
The described resource has a version, edition, or adaption, namely the referenced resource.

This relationship may be used to point to older versions of the resource, to show how the resource has changed over time. For example, if a new resource is added to the aggregation, it may be useful to maintain the old aggregation as a separate version.

  • Name: dcterms:hasVersion
  • URI: http://purl.org/dc/terms/hasVersion

  • Aggregations MAY have versions.
  • Resource Maps MAY have versions.
  • Aggregated Resources MAY have versions.
  • External Resources SHOULD NOT have versions listed.

Motivating Use Cases:
  • Referencing older versions of Resource Maps or Aggregated Resources. For example Arxiv descriptions link to previous versions of the paper.

3. Classes

In order to give a semantic type to an object, we use rdf:type and a class from some vocabulary, as below:

URI-AR1 rdf:type http://purl.org/dc/dcmitype/Collection

The following classes, other than the final two required for audience, are drawn from the DCMI Types Vocabulary and seem to be generic and useful. However, there are of course many other vocabularies with similarly generic and useful types. Suggestions as to more appropriate or useful class heirarchies are encouraged.

5.1 Generic Classes

Collection
An aggregation of resources. A bag, sequence, list, set or other aggregation of multiple resources.
  • Name: dcmitype:Collection
  • URI: http://purl.org/dc/dcmitype/Collection

Dataset
Data encoded in a defined structure.
  • Name: dcmitype:Dataset
  • URI: http://purl.org/dc/dcmitype/Dataset

InteractiveResource
A resource requiring interaction from the user to be understood, executed or experienced.
  • Name: dcmitype:InteractiveResource
  • URI: http://purl.org/dc/dcmitype/InteractiveResource

MovingImage
A series of visual representations imparting an impression of motion when shown in succession.
  • Name: dcmitype:MovingImage
  • URI: http://purl.org/dc/dcmitype/MovingImage
  • SubClassOf: dcmitype:Image

Service
A system that provides one or more functions.
  • Name: dcmitype:Service
  • URI: http://purl.org/dc/dcmitype/Service

Software
A computer program in source or compiled form.
  • Name: dcmitype:Software
  • URI: http://purl.org/dc/dcmitype/Software

Sound
A resource primarily intended to be heard.
  • Name: dcmitype:Sound
  • URI: http://purl.org/dc/dcmitype/Sound

StillImage
A static visual representation other than Text.
  • Name: dcmitype:StillImage
  • URI: http://purl.org/dc/dcmitype/StillImage
  • SubClassOf: dcmitype:Image

Text
A resource consisting primarily of words for reading.
  • Name: dcmitype:Text
  • URI: http://purl.org/dc/dcmitype/Text

Person
The class of 'persons', living, dead, real or imaginary. This should be used, for example, to specify that the intended audience of a resource is a human agent, as opposed to a software agent (see below).
  • Name: foaf:Person
  • SubClassOf: foaf:Agent

SoftwareAgent
A piece of software that takes actions. For example a web crawler or data mining agent.
  • Name: protonkm:SoftwareAgent

5.2 Scholarly Communication Classes

There are many different vocabularies which have tried to define classes for scholarly communication, and choosing between them is non trivial. We as a community should, however, come to some initial decisions in order to demonstrate how other communities can bring their own domain into the OAI-ORE world.

The first three entries in the list above seem to be the most appropriate, followed by the next two should the prove to be taken up in practice. It's not necessary to pick terms from only one vocabulary, but it's certainly easier. We certainly do not need to invent a new one.


Annotated bibliography:

http://ontoware.org/projects/swrc/
"The SWRC (Semantic Web for Research Communities) is an ontology for modeling entities of research communities such as persons, organisations, publications (bibliographic metadata) and their relationships." It seems to contain a good set of classes and relationships, with intuitive names. Seems to be reasonably well used and published about. The OWL was last updated (at time of writing) in June 2007, so seems reasonably current.

http://www.iswc2006.semanticweb.org/projects/semdis/swetodblp/
http://lsdis.cs.uga.edu/~aleman/research/swetodblp/
"SwetoDblp is a large-size ontology (spin-off of SWETO ontology) focused on bibliography data of Computer Science publications where the main data source is DBLP." Again, seems to have all of the classes that we require. The second link above is to a comparison of SwetoDblp and other ontologies. Seems current: recently published about and recently updated (August 2007).

http://www.marcont.org/ontology/documentation/marcont.html
Thankfully, no three digit numbers in sight! Has all of the useful classes and properties. They describe the project: "The goal of MarcOnt bibliographic ontology is to provide a uniform bibliographic description format. It should capture concepts from existing formats such as Bibtex, Dublin Core, MARC21." Recently updated (June 2007) and ongoing work (2004-2007).

http://www.ukoln.ac.uk/repositories/digirep/index/Eprints_Type_Vocabulary_Encoding_Scheme
A "simple vocabulary for use with the dc:type property" for Eprints, from the Dublin Core community. Again, seems to cover the majority of the types, but as of yet hard to assess how much takeup it will have outside of the ePrints/DC area.

http://wiki.bibliontology.com/index.php/Ontology_Working_Draft
http://bibliontology.com/
Bruce D'Arcus and Frédérick Giasson's bibliographic ontology. Quite current, but by no means finished. Seems to be a mpre recent version of Bruce's http://vocab.org/biblio/schema, and I guess is possible this will form the basis of OpenOffice 3.0's bibliographic support. While not finished, it's mostly in need of better documentation; the classes and relationships seem very good.

http://www-ksl.stanford.edu/knowledge-sharing/ontologies/html/bibliographic-data/index.html
A very old (for the web) project from Stanford. Copyright is 1992, but it defines a comprehensive set of classes and could easily be brought into line with modern technologies.

http://zeitkunst.org/bibtex/0.1/
Early version of bibTex in OWL, but doesn't seem to be actively developed beyond 0.1, which has a lot of questions raised in the description. 2004 and version 0.1, so unlikely to be useful other than as a reference.

http://neuroscientific.net/sisc/sisc_introduction.htm
Semantically Interlinked Scientific Communities. Perhaps not directly relevant, but a good demonstration of very similar work in a specific discipline. Note the use of SIOC (Semantically Interlinked Open Communities, http://sioc-project.org/, http://www.w3.org/Submission/2007/02/ ) which seems to have a lot of support.

http://rs.tdwg.org/ontology/voc/PublicationCitation
Another related effort from the Biodiversity discipline, also very current. It would be good to try to reach some sort of cross-disciplinary consensus if possible!

http://www.cs.toronto.edu/semanticweb/maponto/ontologies/Biblio.owl
FRBR derives ontology for bibliographic information. No documentation could be found, so very hard to assess. Still, worthy of a point in the list.

http://www.cs.vu.nl/~pmika/research/burst/BuRST.html
"Bibliographic Management using RSS Technology" (Not sure where the M went?) "... is a lightweight specification for publishing bibliographic information using RSS 1.0". Heh, that sounds kind of familiar, though OAI-ORE isn't tied specifically to bibliographic material and uses ATOM instead of RSS. Uses SWRC for classes etc, but is worth looking over to compare to our approach.
