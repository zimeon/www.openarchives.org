The purpose of this document is to specify a serialization for Resource Maps based on and compliant with the Atom syndication format. This serialization may be referred to as the Resource Map Profile of Atom. This specification is one of several documents comprising the OAI-ORE specification and user guide.

1. Introduction

Open Archives Initiative Object Reuse and Exchange (OAI-ORE) defines standards for the description and exchange of Aggregations of Web resources. OAI-ORE introduces the notion of Resource Maps that describe an Aggregation. A Resource Map is a specialization of a Named Graph [Named Graph], which  is a set of triples named by a URI.  This URI makes it possible for other applications to link to or refer to the Resource Map. A Resource Map identifies an Aggregation, it asserts the finite set of constituent resources (the Aggregated Resources) of the Aggregation, and it can express types and relationships pertaining to the Aggregation and its Aggregated Resources. Each Aggregation may be described by one or more Resource Maps, each of which must have exactly one Representation that is a serialization of the Resource Map according to a specific format. The purpose of this document is to specify a serialization for Resource Maps based on and compliant with the Atom syndication format [RFC4287]. This serialization may be referred to as the Resource Map Profile of Atom. Atom-based Resource Maps may be appealing to a number of application domains because of their simplicity and the existence of standard Web feed technology.

The ORE Model imposes constraints on the types of triples that can be expressed in the Resource Map RDF Graph. Although the hierarchical structure of Atom Feeds does not naturally accomodate the expression of aribtrary graphs, the Atom-based serialization of the Resource Map RDF Graph described in this document is capable of expressing the entire range of triples allowed by the ORE Abstract Data Model in Atom. Moreover, Atom-based Resource Maps can automatically and losslessly be transformed into the RDF/XML serializations of Resource Maps as described in the ORE User Guide - Resource Map Implementation in RDF/XML [ORE User Guide RDF/XML].

This specifcation uses concepts and terms introduced in the following ORE documents:

  • ORE Specification - Abstract Data Model [ORE Model];
  • ORE Specification - Vocabulary [ORE Vocabulary].

Atom representations of Resource Maps are fully compliant with the Atom Syndication Format [RFC4287] and adhere to a specific profile of it:

  • Section 2 defines how entities of the ORE Model relate to entities of the Atom model.
  • Section 3 and Section 4 restrict the use of some Atom constructs and elements, and specify how to use the extensibility features of Atom to address ORE requirements.

The ORE User Guide - Resource Map Implementation in Atom [ORE User Guide Resource Map] provides Atom implementation guidance that supplement this specification.

1.1 Notational Conventions

The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC2119].

This specification uses the same shorthands to terms from the XML Infoset [XML Infoset] as used in [RFC4287]: the phrase "Information Item" is omitted when naming Element Information Items and Attribute Information Items. Therefore, when this specification uses the term "element," it is referring to an Element Information Item in Infoset terms. Likewise, when it uses the term "attribute," it is referring to an Attribute Information Item.

This specification uses the common term URI [RFC3986] to mean both IRI [RFC3987] and URI.

1.3 Compliance

Atom representations of Resource Maps MUST be compliant with the Atom Syndication Format [RFC4287]. Moreover, they MUST adhere to the semantic mapping between the ORE Model and the Atom model detailed Section 2, as well as to the restrictions and extensions for Atom representations of Resource Maps detailed in Section 3 and Section 4.

Specifically, an Atom representation of a Resource Map MUST be a well-formed XML document that is valid according to the Relax NG Compact Schema of [RFC4287]. Moreover, it MUST be valid according to the more specific Schematron Schema of Appendix C that further restricts the Atom syntax to represent Resource Maps.

1.4 Example

Throughout the remainder of this document, the following scenario will be used to illustrate the specifics of Atom-based representations of Resource Maps.

D-Lib Magazine <http://www.dlib.org> decides to adopt the OAI-ORE specification by minting Aggregation URIs (URI-A in the ORE Model) for each of its articles, and by making an Atom-based Resource Map available for each of those Aggregations. Each Aggregation consists of all Web resources that make up a D-Lib article, and each Atom-based Resource Map expresses this constituency in an Atom-compliant manner. The scenario zooms in on the following article:

Joan A. Smith, Frank McCown, Michael L. Nelson. Observed Web Robot Behavior on Decaying Web Subsites. D-Lib Magazine, Volume 12, Number 2, February 2006. doi:10.1045/february2006-smith. <http://dx.doi.org/10.1045/february2006-smith>.

In addition, it makes the Atom-based Resource Map that describes this Aggregation available at another URI (URI-R of a Resource Map in the ORE Model): <http://www.dlib.org/dlib/february06/smith/aggregation.atom>.

2. Correspondence between the ORE Model and the Atom Model

2.1. Mapping between the ORE Model and the Atom Model: Basics

Table 1, Figure 1 and Figure 2 illustrate how entities from the ORE Model map to entities from the Atom model [RFC4287].

In general, the mapping leverages the fact that the Atom model distinguishes between two URIs that pertain to an Atom Feed:

  • The URI of the Atom Feed itself, which corresponds to the URI of an Aggregation (URI-A) of the ORE Model, and
  • The URI from which an Atom Feed Document is retrievable, which corresponds with the URI of a Resource Map (URI-R) that describes an Aggregation, and from which a representation is available that serializes the Resource Map according to a specific format (Atom in this case).

The mapping also introduces an Atom Entry for each Aggregated Resource of the Aggregation. Two URIs introduced by the ORE Model in relation to an Aggregated Resource are expressed in Atom in the following way:

  • The URI of the Aggregated Resource (URI-AR) itself is conveyed in the Atom Entry as the link target of a unique atom:link child element of atom:entry that expresses an "alternate" relationship; this URI is the value of the href attribute of this atom:link.
  • The URI of the Proxy for an Aggregated Resource (URI-P), which is introduced as an optional concept by the ORE Model, corresponds with the identifier of the Atom Entry that conveys the Aggregated Resource. This URI denotes the Aggregated Resource not in its own right, but rather as it exists in the context of the Aggregation. The URI of the Proxy for the Aggregated Resource is the value of the atom:id child element of the atom:entry. Since atom:id are mandatory in Atom, the use of Proxy URIs in the Resource Map Profile of Atom is not optional but REQUIRED.

Furthermore, the mapping introduces an unambiguous way to interpret other elements provided in an Atom-based Resource Map in terms of the ORE Model:

  • For child elements of atom:feed this mapping between Atom and ORE is detailed in Table 2. In summary:
    • The atom:generator, atom:updated, and atom:rights child elements of atom:feed pertain to the Resource Map, and correspond with the dcterms:creator, dcterms:modified and dc:rights elements introduced in the ORE Model as Metadata pertaining to a Resource Map;
    • All other child elements of atom:feed that are in the Atom namespace pertain to the Aggregation described by the Resource Map;
    • The content of an rdf:Description child element of atom:feed pertains to either the Aggregation or the Resource Map, depending on the URI (URI-A or URI-R) that is provided as the value of the rdf:about attribute of the element;
    • The content of additional rdf:Description child elements of atom:feed pertain to the resource for which the URI is provided as the value of the rdf:about attribute of the element and are used to express types and relationships of resources that have a direct or indirect relationsip to the Aggregation or the Resource Map;
    • Other child elements of atom:feed have no correspondence in the ORE Model, and MAY be ignored by ORE processors.
  • For child elements of atom:entry this mapping between Atom and ORE is detailed in Table 3 . In summary:
    • The atom:published, atom:updated, atom:rights and atom:source child elements of atom:entry are Atom-specific metadata. They pertain to the Atom Entry and have no correspondence in the ORE Model;
    • An atom:link child element of atom:entry that has a rel attribute with a value of "via" may be used to express the notion of Lineage introduced in the ORE Model. It pertains to the Proxy of the Aggregated Resource conveyed by the Atom Entry;
    • All other child elements of atom:entry that are in the Atom namespace pertain to the Aggregated Resource conveyed in the Atom Entry;
    • The content of an rdf:Description child element of atom:entry pertains to either the Aggreged Resource or to its Proxy, depending on the URI (URI-AR or URI-P) that is provided as the value of the rdf:about attribute of the element;
    • The content of additional rdf:Description child elements of atom:entry pertain to the resource for which the URI is provided as the value of the rdf:about attribute of the element and are used to express types and relationships of resources that have a direct or indirect relationsip to the Aggregated Resource or to its Proxy;
    • Other child elements of atom:entry have no correspondence in the ORE Model, and MAY be ignored by ORE processors.

3. Use of Atom Constructs in the Resource Map Profile of Atom

This Section specifies the use of common Atom Constructs to represent Resource Maps in Atom. When, in the remainder of this document, an Atom element is identified as being a particular kind of construct, it inherits the requirements specified here.

3.1 Text Constructs

In the Resource Map Profile of Atom:

  • The Text construct MUST be used in a manner described in the Text Construct section of [RFC4287].

3.2 Person Constructs

In the Resource Map Profile of Atom:

  • The Person construct MUST be used in a manner described in the Person Construct section of [RFC4287].

3.3 Date Constructs

In the Resource Map Profile of Atom:

  • The Date construct MUST be used in a manner described in the Date Construct section of [RFC4287].
  • In the Resource Map Profile of Atom, expressing date-times in UTC is RECOMMENDED. The use of the special UTC designator ("Z") is RECOMMENDED over the notation using a numeric time zone offset. Hence, the RECOMMENDED format is YYYY-MM-DDThh:mm:ssZ.

4. Use of Atom Elements in the Resource Map Profile of Atom

4.1 Container Elements

4.1.1 The "atom:feed" Element

In the Resource Map Profile of Atom:

  • The "atom:feed" element has semantics and properties as described in the atom:feed Element section of [RFC4287] unless specified differently below.
  • An Atom Feed corresponds with an Aggregation of the ORE Model, and an Atom Feed Document corresponds with the (Atom-based) Resource Map of the ORE Model.
  • The following rules apply regarding child elements of the atom:feed element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MAY contain any number atom:author elements.
      • The atom:author element SHOULD convey meaningful authorship information for the Aggregation.
      • The use of a URI to identify an author is RECOMMENDED.
    • atom:feed elements MUST contain exactly one atom:category element typing the Atom Feed as an Aggregation.
      • This element MUST have a scheme attribute value of "http://www.openarchives.org/ore/terms/", a term attribute value of "http://www.openarchives.org/ore/terms/Aggregation", and a label attribute value of "Aggregation".
    • atom:feed elements MAY contain any number of atom:category elements in addition to the aforementioned one.
      • The atom:category element MUST convey a categorization or semantic type information for the Aggregation; It corresponds with the rdf:type relationship of the ORE Model.
    • atom:feed elements MAY contain any number of atom:contributor elements.
      • The atom:contributor element SHOULD convey meaningful contributor information for the Aggregation.
      • The use of a URI to identify a contributor is RECOMMENDED.
    • atom:feed elements MUST contain one or more atom:entry elements.
      • Each atom:entry element MUST convey an Aggregated Resource of the Aggregation.
    • atom:feed elements MUST contain exactly one atom:generator element.
      • The atom:generator element MUST convey the author of the Atom-based Resource Map; it corresponds with the dcterms:creator element introduced in the ORE Model as Metadata pertaining to a Resource Map.
      • The uri attribute of this atom:generator element is REQUIRED, and MUST convey a URI associated with the author of the Atom-based Resource Map.
    • atom:feed elements MUST NOT contain more than one atom:icon element.
      • The atom:icon element conveys an icon for the Aggregation.
    • atom:feed elements MUST contain exactly one atom:id element.
      • The atom:id element MUST convey the URI of the Aggregation (URI-A) of the ORE Model.
      • Guidelines on minting URIs for Aggregations are provided in the ORE User Guide - HTTP Implementation and Multiple Serializations.
    • atom:feed elements MUST contain one atom:link element with a rel attribute value of "self".
      • The href attribute value of this atom:link element MUST be the URI of the (Atom-based) Resource Map (URI-R) of the ORE Model.
      • The type attribute value of this atom:link element SHOULD be "application/atom+xml".
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "related".
      • The href attribute value of this atom:link element MUST be a URI of a resource that is similar to the Aggregation described by the Resource Map. It corresponds with the ore:similarTo relationship of the ORE Model.
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "alternate".
      • The href attribute value of this atom:link element MUST be the URI of another Resource Map that describes the same Aggregation as the one described by the current Atom-based Resource Map, i.e. the linked Resource Map and the current Resource Map MUST have the same URI-A. This atom:link corresponds with the ore:isDescribedBy relationship introduced in the ORE Model.
      • The type attribute value of this link SHOULD convey the media type of the other Resource Map.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "license".
      • The href attribute value of this atom:link element MUST be the URI of a rights expression pertaining to the Aggregation.
      • It is RECOMMENDED that the referenced rights expression is machine readable.
    • atom:feed elements MAY contain additional atom:link elements beyond those described above.
      • These atom:link elements have no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "alternate" that has the same combination of type and hreflang attribute values.
    • atom:feed elements MUST NOT contain more than one atom:logo element.
      • The atom:logo element conveys a logo for the Aggregation.
      • The atom:logo element has no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • atom:feed elements MUST NOT contain more than one atom:rights element.
      • The atom:rights element SHOULD NOT be used to convey machine-readable licensing information.
      • The atom:rights element MUST convey the rights pertaining to the Resource Map; it corresponds with the dc:rights element introduced in the ORE Model as Metadata pertaining to a Resource Map.
    • atom:feed elements MUST NOT contain more than one atom:subtitle element.
      • The atom:subtitle element SHOULD convey a meaningful subtitle for the Aggregation.
    • atom:feed elements MUST contain exactly one atom:title element.
      • The atom:title element SHOULD convey a meaningful title for the Aggregation.
    • atom:feed elements MUST contain exactly one atom:updated element.
      • The atom:updated element MUST convey the date-time of most recent update of the Atom-based Resource Map; it corresponds with the dcterms:modified element introduced in the ORE Model as Metadata pertaining to a Resource Map.
    • atom:feed elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Aggregation (URI-A).
      • This element MUST be used to convey types and relationships pertaining to the Aggregation that can not be expressed using child elements of atom:feed that are in the Atom namespace.
    • atom:feed elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Atom-based Resource Map (URI-R).
      • This element MUST be used to convey types and relationships pertaining to the Atom-based Resource Map that can not be expressed using child elements of atom:feed that are in the Atom namespace.
    • atom:feed elements MAY contain any number of rdf:Description elements with a value of their rdf:about attribute equal to a URI other than the URI of the Aggregation (URI-A) or the Atom-based Resource Map (URI-R).
      • Such elements MAY be used to convey types and relationships pertaining to resources other than the Aggregated Resources that have a direct or indirect relationships with the Aggregation or the Resource Map.
    • Child elements of atom:feed other than the ones listed above have no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • An Atom Feed Document MUST NOT have multiple atom:entry elements with the same atom:id value, as is permitted by [RFC4287]. Such entries would convey the same Aggregated Resource, and might contain conflicting assertions about that Aggregated Resource within the confines of the same Resource Map. As a result, updating an atom:entry in the Resource Map Profile of Atom MUST always be achieved by editing the existing atom:entry and by changing the value of its atom:updated child element. It MUST NOT be achieved by adding an additional atom:entry with the same atom:id as the already existing one.

4.1.2 The "atom:entry" Element

In the Resource Map Profile of Atom:

  • The "atom:entry" element has semantics and properties as described in the atom:entry Element section of [RFC4287] unless specified differently below.
  • An Atom entry conveys an Aggregated Resource, Types and Relationships pertaining to the Aggregated Resource, as well as some Atom-specific information.
  • The following rules apply regarding child elements of the atom:entry element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:entry elements MAY contain any number of atom:author elements.
      • The atom:author element SHOULD convey a meaningful author for the Aggregated Resource.
      • The use of a URI to identify an author is RECOMMENDED.
      • If the author of the Aggregated Resource is considered to be anonymous or otherwise unknown, an atom:author element containing nothing except an empty atom:name child element MUST be provided.
      • If no atom:author element is provided, then the atom:author elements of the contained atom:source element are considered to apply.
      • If none of these atom:author elements exist, then the atom:author elements of the atom:feed element are considered to apply.
    • atom:entry elements MUST NOT contain more than one atom:category element that types the Aggregated Resource as an Aggregation. Typing an Aggregated Resource in this manner corresponds with the notion of Nesting Aggregations introduced in the ORE Model.
      • This element MUST have a scheme attribute value of "http://www.openarchives.org/ore/terms/", a term attribute value of "http://www.openarchives.org/ore/terms/Aggregation", and SHOULD have a label attribute value of "Aggregation".
    • atom:entry elements MAY contain any number of additional atom:category elements.
      • The atom:category element MUST convey a categorization or semantic type information for the Aggregated Resource; It corresponds with the rdf:type relationship of the ORE Model.
    • atom:entry elements MUST NOT contain any atom:content elements.
    • atom:entry elements MAY contain any number of atom:contributor elements.
      • The atom:contributor element SHOULD convey meaningful contributor information for the Aggregated Resource.
      • The use of a URI to identify a contributor is RECOMMENDED.
    • atom:entry elements MUST contain exactly one atom:id element.
      • The atom:id element MUST convey the URI of the Proxy for the Aggregated Resource (URI-P) as introduced in the ORE Model.
      • Guidelines on minting URIs for Proxies are provided in the ORE User Guide - HTTP Implementation and Mutliple Serializations.
    • atom:entry elements MUST contain one atom:link element with a rel attribute value of "alternate".
      • The href attribute value of this atom:link element MUST be the URI of the Aggregated Resource (URI-AR).
    • atom:entry elements MAY contain any number of atom:link elements with a rel attribute value of "related".
      • The href attribute value of this atom:link element MUST be the URI of another Aggregation in which the Aggregated Resource is aggregated. It corresponds to the ore:isAggregatedBy relationship introduced in the ORE Model.
      • The type and length attributes of this atom:link SHOULD NOT be used as Aggregations are non-information resources; they have no mime type nor lenght.
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "self".
      • This is the (optional) preferred URI for retrieving Atom Entry Documents representing this Atom entry.
      • The type attribute value of this atom:link element SHOULD be "application/atom+xml".
      • The href attribute value of this atom:link element MUST NOT be the URI of the Aggregated Resource (URI-AR).
      • This atom:link element has no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • atom:entry elements MAY contain any number of atom:link elements with a rel attribute value of "via".
      • This link MUST be used to express the lineage of the Aggregated Resource in another Aggregation as introduced in the ORE Model.
      • The href attribute value of this atom:link element MUST be the URI of a Proxy for the Aggregated Resource in another Aggregation.
    • atom:entry elements MUST NOT contain any atom:link elements other than the aformentioned ones.
    • atom:entry elements MUST NOT contain more than one atom:published element.
      • The atom:published element SHOULD convey the date-time of creation of the atom:entry element that conveys the Aggregated Resource. In the ORE perspective, it corresponds with the date-time the Aggregated Resource was first aggregated in the Aggregation.
    • atom:entry elements MUST NOT contain more than one atom:rights elements.
      • The atom:rights element SHOULD NOT be used to convey machine-readable licensing information.
      • The atom:rights element MUST convey the rights pertaining to the atom:entry;
      • If no atom:rights element is provided, then the atom:rights of the atom:feed element are considered to apply.
    • atom:entry elements MUST NOT contain more than one atom:source element.
      • The atom:source element MUST convey the origin of the atom:entry in another Atom Feed Document;
      • The atom:source element has no correspondence in the ORE Model.
    • atom:entry elements MUST NOT contain more than one atom:summary element.
      • The atom:summary element SHOULD convey meaningful summary for the Aggregated Resource.
    • atom:entry elements MUST contain exactly one atom:title element.
      • The atom:title element SHOULD convey a meaningful title for the Aggregated Resource.
    • atom:entry elements MUST contain exactly one atom:updated element.
      • The atom:updated element SHOULD convey the date-time of most recent update of the atom:entry.
    • atom:entry elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Aggregated Resource (URI-AR).
      • This element MUST be used to convey types and relationships pertaining to the Aggregated Resource that can not be expressed using child elements of atom:entry that are in the Atom namespace.
    • atom:entry elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Proxy for the Aggregated Resource (URI-P).
      • This element MUST be used to convey types and relationships pertaining to the Proxy for the Aggregated Resource.
    • atom:entry elements MAY contain any number of rdf:Description elements with a value of their rdf:about attribute equal to a URI other than the URI of the Aggregated Resource (URI-AR) or of its Proxy (URI-P).
      • Such elements MAY be used to convey types and relationships pertaining to resources (other than the Aggregation and the Resource Map) that have a direct or indirect relationships with the Aggregated Resource.
    • Child elements of atom:entry other than the ones listed above have no correspondence in the ORE Model, and MAY be ignored by ORE processors.

4.2.1 The "atom:author" Element

In the Resource Map Profile of Atom:

  • The "atom:author" element has semantics and properties as described in the atom:author Element section of [RFC4287] unless specified differently below.
  • The "atom:author" element is a Person construct that indicates the author of an Atom Feed or Atom Entry.
  • The following rules apply regarding the atom:author element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MAY contain any number of atom:author elements.
      • The atom:author element SHOULD convey meaningful authorship information for the Aggregation.
      • The use of a URI to identify an author is RECOMMENDED.
    • atom:entry elements MAY contain any number of atom:author elements.
      • The atom:author element SHOULD convey a meaningful author for the Aggregated Resource.
      • The use of a URI to identify an author is RECOMMENDED.
      • If the author of the Aggregated Resource is considered to be anonymous or otherwise unknown, an atom:author element containing nothing except an empty atom:name child element MUST be provided.
      • If no atom:author element is provided, then the atom:author elements of the contained atom:source element are considered to apply.
      • If none of these atom:author elements exist, then the atom:author elements of the atom:feed element are considered to apply.

In the D-Lib Magazine example, Michael Nelson, Joan Smith, and Frank McCown are authors of the journal article for which the Aggregation is created. In order to convey meaningful authorship information for the Aggregation, they are listed as authors of the Aggregation. Note that this authorship is automatically inherited for all Aggregated Resources conveyed in the Atom-based Resource Map, unless authorship of individual Aggregated Resources is specified otherwise. To express authorship of an Aggregated Resource, the atom:author child element of the atom:entry that conveys the Aggregated Resource is used. If the authorship of an Aggregated Resource is unknown, and should not be inherited from the Aggregation, an empty atom:author element must be provided for the atom:entry that conveys the Aggregated Resource.

        <atom:uri>http://www.joanasmith.com/</atom:uri>
    </atom:author>
    <atom:author>

        <atom:uri>http://www.cs.odu.edu/~fmccown/</atom:uri>

    <!-- Aggregated Resources -->

      • The atom:category element MUST convey a categorization or semantic type information for the Aggregated Resource; It corresponds with the rdf:type relationship of the ORE Model.

4.2.6 The "atom:id" Element

In the Resource Map Profile of Atom:

  • The "atom:id" element has semantics and properties as described in the atom:id Element section of [RFC4287] unless specified differently below.
  • The "atom:id" element conveys a permanent, universally unique identifier for an Atom feed or Atom entry.
  • The normalization guidelines for generating atom:id elements are as described in the atom:id Element section of [RFC4287].
  • The rules for comparing atom:id are as expressed in the Comparing atom:id section of [RFC4287].
  • The following rules apply regarding the atom:id element; rules that are more specific than the ones of [RFC4287] are shown in italic:

    • atom:feed elements MUST contain exactly one atom:id element.
      • The atom:id element MUST convey the URI of the Aggregation (URI-A) of the ORE Model.
      • Guidelines on minting URIs for Aggregations are provided in the ORE User Guide - HTTP Implementation and Multiple Serializations.
    • atom:entry elements MUST contain exactly one atom:id element.
      • The atom:id element MUST convey the URI of the Proxy for the Aggregated Resource (URI-P) as introduced in the ORE Model.
      • Guidelines on minting URIs for Proxies are provided in the ORE User Guide - HTTP Implementation and Mutliple Serializations.

In the D-Lib Magazine example, <http://www.dlib.org/dlib/february06/smith/aggregation> is the URI of the Aggregation (URI-A); it is conveyed as the content of the Feed-level atom:id element. The example also shows an Aggregated Resource of this Aggregation. Its URI (URI-AR) is provided in an Entry-level atom:link element that has a value of "alternate" for its rel attribute: <http://www.dlib.org/dlib/february06/smith/02smith.html>. The URI provided as the content of the Entry-level atom:id element <http://oreproxy.org/r?what=http://www.dlib.org/dlib/february06/smith/pg1-13.html&where=http://www.dlib.org/dlib/february06/smith/aggregation> uniquely identifies the Atom Entry, and is the URI of the Proxy (URI-P) for the Aggregated Resource.

4.2.7 The "atom:link" Element

In the Resource Map Profile of Atom:

  • The "atom:link" element has semantics and properties as described in the atom:link Element section of [RFC4287] unless specified differently below.
  • The atom:link element has attributes as described in the sections The "href" Attribute, The "rel" Attribute, The "type" Attribute, The "hreflang" Attribute, The "title" Attribute and The "length" Attribute of [RFC4287].
  • The following rules apply regarding the atom:link element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST contain one atom:link element with a rel attribute value of "self".
      • The type attribute value of this atom:link element SHOULD be "application/atom+xml".
      • The href attribute value of this atom:link element MUST be the URI of the Atom-based Resource Map (URI-R) of the ORE Model.
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "related".
      • The href attribute value of this atom:link element MUST be a URI of a resource that is similar to the Aggregation described by the Resource Map. It corresponds with the ore:similarTo relationship of the ORE Model.
    • atom:feed elements MAY contain any number of atom:link elements with a rel attribute value of "alternate".
      • The href attribute value of this atom:link element MUST be the URI of another Resource Map that describes the same Aggregation as the one described by the current Atom-based Resource Map, i.e. the linked Resource Map and the current Resource Map MUST have the same URI-A. This atom:link corresponds with the ore:isDescribedBy relationship introduced in the ORE Model.
      • The type attribute value of this link SHOULD convey the media type of the other Resource Map.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "license".
      • The href attribute value of this atom:link element MUST be the URI of a rights expression pertaining to the Aggregation.
      • It is RECOMMENDED that the referenced rights expression is machine readable.
    • atom:feed elements MAY contain additional atom:link elements beyond those described above.
      • These atom:link elements have no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • atom:feed elements MUST NOT contain more than one atom:link element with a rel attribute value of "alternate" that has the same combination of type and hreflang attribute values.
    • atom:entry elements MUST contain exactly one atom:id element.
      • The atom:id element MUST convey the URI of the Proxy for the Aggregated Resource (URI-P) as introduced in the ORE Model.
      • Guidelines on minting URIs for Proxies are provided in the ORE User Guide - HTTP Implementation and Mutliple Serializations.
    • atom:entry elements MUST contain one atom:link element with a rel attribute value of "alternate".
      • The href attribute value of this atom:link element MUST be the URI of the Aggregated Resource (URI-AR).
    • atom:entry elements MAY contain any number of atom:link elements with a rel attribute value of "related".
      • The href attribute value of this atom:link element MUST be the URI of another Aggregation in which the Aggregated Resource is aggregated. It corresponds to the ore:isAggregatedBy relationship introduced in the ORE Model.
      • The type and length attributes of this atom:link SHOULD NOT be used as Aggregations are non-information resources; they have no mime type nor lenght.
    • atom:entry elements MUST NOT contain more than one atom:link element with a rel attribute value of "self".
      • This is the (optional) preferred URI for retrieving Atom Entry Documents representing this Atom entry.
      • The type attribute value of this atom:link element SHOULD be "application/atom+xml".
      • The href attribute value of this atom:link element MUST NOT be the URI of the Aggregated Resource (URI-AR).
      • This atom:link element has no correspondence in the ORE Model, and MAY be ignored by ORE processors.
    • atom:entry elements MAY contain any number of atom:link elements with a rel attribute value of "via".
      • This link MUST be used to express the lineage of the Aggregated Resource in another Aggregation as introduced in the ORE Model.
      • The href attribute value of this atom:link element MUST be the URI of a Proxy for the Aggregated Resource in another Aggregation.
    • atom:entry elements MUST NOT contain any atom:link elements other than the aformentioned ones.

In the D-Lib Magazine example, the URI of the Atom-based Resource Map (URI-R) that describes the Aggregation is provided in a Feed-level atom:link element that has a value of "self" for its rel attribute: <http://www.dlib.org/dlib/february06/smith/aggregation.atom>. Note that the URI of the Aggregation (URI-A) itself is conveyed as the content of the Feed-level atom:id element: <http://www.dlib.org/dlib/february06/smith/aggregation>. Other URIs (e.g. a DOI) that are associated with the Aggregation are provided using an atom:link element with a rel attribute value of "related" and a href attribute value set to another URI associated with the Aggregation: <info:doi/10.1045/february2006-smith>. A pointer to a Resource Map that describes the Aggregation in a format other than Atom is provided in a Feed-level atom:link element that has a value of "alternate" for its rel attribute: <http://www.dlib.org/dlib/february06/smith/aggregation.rdf> is the URI of a Resource Map (URI-R) that describes the Aggregation in an RDF/XML serialization. An additional Feed-level atom:link element conveys the rights pertaining to the Aggregation: the URI <http://creativecommons.org/licenses/by-nc/2.5/rdf> references a machine-readable rights description that pertains to the Aggregation. This URI is provided as the value of the href attribute of a Feed-level atom:link element that has a value of "license" for its rel attribute.

The atom:entry shown in the example conveys the Aggregated Resource with URI <http://www.dlib.org/dlib/february06/smith/MLN_Google.png>. This URI is provided as the value of the href attribute of the Entry-level atom:link element that has a value of "alternate" for its rel attribute. An additional Entry-level atom:link element that has a value of "via" for its rel attribute expresses the lineage of the Aggregated Resource in another Aggregation. The value of the href attribute of this atom:link is the URI of the Proxy for the Aggregated Resource in another Aggregation: <http://oreproxy.org/r?what=http://www.dlib.org/dlib/february06/smith/MLN_Google.png&where=http://www.example.org/aggregation/45887>.

4.2.11 The "atom:source" Element

  • The "atom:source" element has no correspondence in the ORE Model; it is distinct from the notion of Lineage of an Aggregated Resource in another Aggregation introduced in the ORE Model.
  • The following rules apply regarding the atom:source element; rules that are more specific than the ones of [RFC4287] are shown in italic:

This is achieved by adding an atom:source child element, if it is not already present, to the atom:entry and including some or all of the origin Atom Feed Document's metadata elements as child elements of the atom:source element.
    • The following child elements of the origin Atom Feed Document's (Resource Map'Document) atom:feed element SHOULD be preserved in this manner: atom:author, atom:id, atom:generator, atom:link, atom:title, atom:rights, atom:category, and atom:updated.

4.2.12 The "atom:subtitle" Element

In the Resource Map Profile of Atom:

  • The "atom:subtitle" element has semantics and properties as described in the atom:subtitle Element section of [RFC4287] unless specified differently below.
  • The "atom:subtitle" is a Text construct that conveys a human-readable description or subtitle for an Atom Feed.
  • The following rules apply regarding the atom:subtitle element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST NOT contain more than one atom:subtitle element.
    • The atom:subtitle element SHOULD convey a meaningful subtitle for the Aggregation.

4.2.13 The "atom:summary" Element

In the Resource Map Profile of Atom:

  • The "atom:summary" element has semantics and properties as described in the atom:summary Element section of [RFC4287] unless specified differently below.
  • The "atom:summary" is a Text construct that conveys a short summary, abstract, or excerpt of an Atom entry.
  • The following rules apply regarding the atom:summary element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:entry elements MUST NOT contain more than one atom:summary element.
    • The atom:summary element SHOULD convey meaningful summary for the Aggregated Resource.

4.2.14 The "atom:title" Element

In the Resource Map Profile of Atom:

  • The "atom:title" element has semantics and properties as described in the atom:title Element section of [RFC4287] unless specified differently below.
  • The "atom:title" is a Text construct that conveys a human-readable title for an Atom feed (Resource Map) or Atom entry.
  • The following rules apply regarding the atom:title element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST contain exactly one atom:title element.
      • The atom:title element SHOULD convey a meaningful title for the Aggregation.
    • atom:entry elements MUST contain exactly one atom:title element.
      • The atom:title element SHOULD convey a meaningful title for the Aggregated Resource.

4.2.15 The "atom:updated" Element

In the Resource Map Profile of Atom:

  • The "atom:updated" element has semantics and properties as described in the atom:updated Element section of [RFC4287] unless specified differently below.
  • The "atom:updated" element is a Date construct indicating the most recent instant in time when an Atom Feed or Atom Entry was modified in a way the authoring authority considers significant. Therefore, not all modifications necessarily result in a changed atom:updated value.
  • The "atom:updated" element, when used as a child element of an Atom feed corresponds to the last modification date-time of the Resource Map as defined in the ORE Model.
  • The "atom:updated" element, when used as a child element of an Atom entry, has no correspondence in the ORE Model.
  • The following rules apply regarding the atom:updated element; rules that are more specific than the ones of [RFC4287] are shown in italic:
    • atom:feed elements MUST contain exactly one atom:updated element.
      • The atom:updated element MUST convey the date-time of most recent update of the Atom-based Resource Map; it corresponds with the dcterms:modified element introduced in the ORE Model as Metadata pertaining to a Resource Map.
    • atom:entry elements MUST contain exactly one atom:updated element.
      • The atom:updated element SHOULD convey the date-time of most recent update of the atom:entry element that conveys the Aggregated Resource.
    • The content of atom:updated SHOULD be set to a new value when relevant changes are made to an atom:feed or an atom:entry. Examples of such relevant changes MAY include the following:
      • the aggregation of an additional resource (i.e. the creation of a new atom:entry that conveys an additional Aggregated Resource);
      • the removal of an Aggregated Resource (i.e. deletion of the atom:entry that conveys an Aggregated Resource) ;
      • the addition or deletion of relationships or types pertaining to the Aggregation, the Resource Map, or to an Aggregated Resource;
      • the addition of a URI of a resource that is similar to the Aggregation;
      • the update of an Aggregated Resource;
    • The content of atom:updated child element of atom:feed MUST be set to a new value if the atom:updated child element of one of its atom:entry elements is set to a new value.

In the D-Lib Magazine example, an Aggregation for the article was first published on August 17th of 2007, more than a year after the publication of the journal article itself. This date of publication of the Resource Map can be observed in the atom:published element of the atom:entry elements: "2007-08-17T21:12:44Z". This date-time also shows up as the value of the atom:updated element of first shown atom:entry, indicating no changes occurred to it since its publication. However, on September 22nd of 2007, the authors of the article requested to update an image used in their paper; that image is the Aggregated Resource conveyed by the second atom:entry that is shown. As a result, D-Lib Magazine decided to change the date-time of the associated atom:updated element to "2007-09-22T07:11:09Z", leaving the date-time of the associated atom:published element unchanged. The change in the atom:updated element of this one atom:entry caused D-Lib Magazine to also change the date-time of the atom:updated element of the atom:feed.

4.2.16 Use of Atom Extension Elements in the Resource Map Profile of Atom

This Section specifies how the Resource Map Profile of Atom makes use of the extension elements allowed by [RFC4287]. The use of extension elements of an atom:feed element are discussed in Section 4.2.16.1, and extension elements of an atom:entry element are discussed in Section 4.2.16.2.

4.2.16.1 Extension Elements for the "atom:feed" Element

The Resource Map Profile of Atom allows the use of OPTIONAL extension elements for the atom:feed element to convey Relationships and Types pertaining to the Aggregation and the Resource Map or pertaining to resources (other than the Aggregated Resources) that have a direct or indirect relationship with them. These Relationships and Types are conveyed in rdf:Description child elements of atom:feed. A recommendation regarding the RDF/XML style to be used for these rdf:Description elements is provided in [ORE User Guide RDF/XML].

In the Resource Map Profile of Atom:

  • atom:feed elements MAY have any number of child elements from namespaces other than the Atom namespace.
  • atom:feed elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Aggregation (URI-A).
    • This element MUST be used to convey types and relationships pertaining to the Aggregation that can not be expressed using child elements of atom:feed that are in the Atom namespace.
  • atom:feed elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Atom-based Resource Map (URI-R).
    • This element MUST be used to convey types and relationships pertaining to the Atom-based Resource Map that can not be expressed using child elements of atom:feed that are in the Atom namespace.
  • atom:feed elements MAY contain any number of rdf:Description elements with a value of their rdf:about attribute equal to a URI other than the URI of the Aggregation (URI-A) or the Atom-based Resource Map (URI-R).
    • Such elements MAY be used to convey types and relationships pertaining to resources other than the Aggregated Resources that have a direct or indirect relationships with the Aggregation or the Resource Map.
  • Child elements of atom:feed other than those in the Atom namespace and the ones listed above have no correspondence in the ORE Model, and MAY be ignored by ORE processors.

In the D-Lib Magazine example, the first rdf:Description child element of atom:feed expresses that the Aggregation is part of D-Lib Magazine. The second rdf:Description states that D-Lib Magazine is a journal.

<?xml version="1.0" encoding="UTF-8" ?>
<atom:feed  xmlns:dcterms="http://purl.org/dc/terms/"
    xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"
    xmlns:atom="http://www.w3.org/2005/Atom"
    xmlns:ore="http://www.openarchives.org/ore/terms/" > 

    <!-- About the Aggregation --> 

    <rdf:Description rdf:about="http://www.dlib.org/dlib/february06/smith/aggregation">
        <dcterms:isPartOf rdf:resource="http://www.dlib.org"/>
    </rdf:Description>
    <rdf:Description rdf:about="http://www.dlib.org">
        <rdf:type rdf:resource="http://purl.org/ontology/bibo/journal"/>
    </rdf:Description>

</atom:feed>
4.2.16.2 Extension Elements for the "atom:entry" Element

The Resource Map Profile of Atom allows the use of OPTIONAL extension elements for the atom:entry element to express Relationships and Types pertaining to the Aggregated Resource conveyed by the atom:entry or pertaing to the Proxy of the Aggregated Resource, as well as to express Relationships and Types pertaining to resources that have a direct or indirect relationship with the Aggregated Resource or its Proxy. These Relationships and Types are conveyed in rdf:Description child elements of atom:entry. A recommendation regarding the RDF/XML style to be used for these rdf:Description elements is provided in the [ORE User Guide RDF/XML].

In the Resource Map Profile of Atom:

  • atom:entry elements MAY have any number of child elements from namespaces other than the Atom namespace.
  • atom:entry elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Aggregated Resource (URI-AR).
    • This element MUST be used to convey types and relationships pertaining to the Aggregated Resource that can not be expressed using child elements of atom:entry that are in the Atom namespace.
  • atom:entry elements SHOULD NOT contain more than one rdf:Description element with a value of its rdf:about attribute equal to the URI of the Proxy for the Aggregated Resource (URI-P).
    • This element MUST be used to convey types and relationships pertaining to the Proxy for the Aggregated Resource.
  • atom:entry elements MAY contain any number of rdf:Description elements with a value of their rdf:about attribute equal to a URI other than the URI of the Aggregated Resource (URI-AR) or of its Proxy (URI-P).
    • Such elements MAY be used to convey types and relationships pertaining to resources (other than the Aggregation and the Resource Map) that have a direct or indirect relationships with the Aggregated Resource.
  • Child elements of atom:entry other than those in the Atom namespace and the ones listed above have no correspondence in the ORE Model, and MAY be ignored by ORE processors.

In the D-Lib Magazine example, the first atom:entry shown has an rdf:Description element that is about the Aggregated Resource conveyed by the atom:entry: <http://www.dlib.org/dlib/february06/smith/02smith.html>. It conveys the date of creation of the Aggregated Resources, and it conveys resources referenced by it. Two additional rdf:Description elements express the types of two of these referenced resources: one is a working paper (also known as pre-print), the other a journal article. In the second atom:entry that is shown, an rdf:Description element indicates that the Aggregated Resource conveyed by the atom:entry, i.e. <http://www.dlib.org/dlib/february06/smith/pg1-13.html> is also available in another format, namely <http://www.dlib.org/dlib/february06/smith/pg1-13.pdf>.

D. GRDDL crosswalk from Atom XML to RDF/XML

The up-to-date version of the GRDDL-compliant XSLT to transform an Atom-based serialization of a Resource Map into a serialization in RDF/XML is available at http://www.openarchives.org/ore/0.9/atom-grddl.xsl. This XSLT uses XSLT 2.0 functionality. In order to allow Atom-based Resource Maps to be readily transformed in this manner, it is RECOMMENDED to add GRDDL namespace and transformation information as attributes to the atom:feed element, as shown below.

<?xml version="1.0" encoding="UTF-8"?>
<atom:feed xmlns:atom="http://www.w3.org/2005/Atom"
      xmlns:grddl="http://www.w3.org/2003/g/data-view#"
      grddl:transformation="http://www.openarchives.org/ore/atom/atom-grddl.xsl">
...
